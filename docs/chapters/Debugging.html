Debugging & Performance

Step-by-step checklists to diagnose and fix slow UI loading, unresponsive pages, and NetLogic bottlenecks in FTOptix projects.

ℹ️ Work through each phase in order. Tick off each item as you verify or fix it. The most impactful checks are marked Critical. Sources: RA Performance Optimization docs and NetLogic Lifecycle CheatSheet.

Phase 1

Measure First — Get Baseline Data

  • Check the FTOptix Runtime Logs Open the runtime log output. Look for Start method on <NodeName> took too long to execute warnings — each is a hard bottleneck. Note every node name reported. Open Runtime Logs Parser
  • Check project-level node count In FTOptix Studio: Tools → Project Information. Record total node count. High node counts directly increase startup and page-load times. Project Information docs
  • Use the Nodes Counter Dialog on the slow page Add the Nodes Counter Dialog widget (Library → Widgets) to the page. Link AliasNode to the page or PanelLoader. At runtime, open the dialog and note the node count for the affected page. If you link to a NavigationPanel, the counter updates on every navigation. Nodes Counter Dialog docs
  • Isolate which page is slow Navigate to each page individually and time how long the UI becomes responsive. This identifies the culprit before you start optimizing.
Phase 2 · Critical

NetLogic Start() — Blocking the Behavior Thread

⚠️ All Start() methods run sequentially on a single behavior thread. A single slow Start() blocks every other NetLogic that follows it in the queue. On a page with 10 NetLogics each blocking for 300 ms, total startup delay = 3 seconds.
  • Identify all NetLogics on the slow page In the Project tree, expand the affected page and list every NetLogic node. They start deepest-first (most nested starts first, page-level starts last).
  • Check for blocking operations inside Start() In each NetLogic, review Start() for: file I/O, database queries, network calls, Thread.Sleep(), long loops, or synchronous waits. Any of these block the entire behavior thread.
  • Move heavy initialization to LongRunningTask Replace blocking code in Start() with a LongRunningTask. Register observers and set up fast state immediately; defer slow work:
    initTask = new LongRunningTask(HeavyInit, LogicObject); initTask.Start();     NetLogic Lifecycle — LongRunningTask pattern
  • Check for node tree modifications inside Start() (Runtime NetLogics) A Runtime NetLogic Start() must NOT add or remove nodes from the page tree. Doing so causes unpredictable loading behavior. Move dynamic tree modifications to the page type definition or use a DelayedTask.
  • Count total NetLogics on the page — reduce if excessive Even if each Start() is fast, many NetLogics accumulate overhead. Consolidate logic into fewer NetLogics where possible.
Phase 3 · Critical

Dynamic Links & Node Count

⚠️ Each Dynamic Link adds at least +3 nodes. A Complex Dynamic Link (with expression converters, key-value maps) adds far more. Node count directly determines page load time.
  • Identify and count all DynamicLinks on the slow page In FTOptix Studio, expand the page tree or use Find in Project to search for DynamicLink nodes. Note how many exist — a page with hundreds of DynamicLinks is a key suspect.
  • Identify Complex DynamicLinks (converters, expressions) Complex DynamicLinks with expression-based converters, key-value converters, or nested links multiply node count significantly. List every Complex DynamicLink and assess whether the converter is truly needed at runtime. Model complexity docs
  • Replace design-time-only DynamicLinks with static values If a DynamicLink only configures a property at design time (the value never changes at runtime), remove the link and set the property to a static value. The link is redundant and wastes nodes.
  • Replace Complex DynamicLinks with Stylesheets where possible Use Stylesheets to drive appearance (color, state styles) instead of per-property Complex DynamicLinks. A single DynamicLink to a stylesheet replaces many individual property links — drastically reducing node count. Stylesheets docs
  • Replace Complex DynamicLinks with NetLogic scripts (central processing) When multiple objects share transformation logic, move the calculation into a NetLogic script that populates a variable. Bind objects to that variable with a simple DynamicLink instead of giving each object its own Complex DynamicLink.
  • Aggregate shared variables using Enumerations or Converters If multiple objects need the same transformed value, create one shared variable or enumeration and link all objects to it — instead of replicating the converter on each object.
Phase 4

UI Object Structure & Hierarchy

  • Check for excessive nesting depth Deep object hierarchies increase node traversal time. Flatten the tree where objects can be placed at a shallower level without losing layout requirements.
  • Remove redundant or empty containers Every Panel or Group object adds nodes even when it contains only one child. Merge containers that serve no structural purpose. UI objects best practices
  • Verify reusable Types (not inline copies) Ensure repeated widgets (buttons, indicators) are defined as a Type and instantiated — not duplicated inline. Instances inherit properties from the type without materializing them, so each instance carries only its overrides rather than a full copy of every property. Optimize Types docs
  • Minimize optional overridden properties on instances Each property overridden on an instance materializes an extra node. Keep instance-level overrides to the minimum required; let inherited values from the Type do the rest.
  • Check for heavy widgets (DataGrid, large Trends) Heavy widgets (DataGrids populating from a database, Trend controls with large datasets) are rendered on the UI thread. Multiple heavy widgets on one page cause cascading delays before NetLogics start. Consider lazy-loading or pagination.
Phase 5

Graphic Rendering & Assets

  • Prefer SVG over PNG for icons and graphics SVG is vector-based (resolution-independent) and can be dynamically tinted or resized at runtime. Use SVG for all icons, symbols, and diagrams. Reserve PNG for photographic content only. Graphic rendering docs
  • Compress PNG images used in the project Large or uncompressed PNGs slow rasterization. Run all PNGs through a compression tool (e.g., TinyPNG, oxipng) before importing them. Check the MediaLibrary folder for any file above 200 KB.
  • Audit SVG complexity Complex SVGs with excessive paths, filters, or gradients also impact render time. Simplify SVGs exported from drawing tools — remove hidden layers, reduce path segments, and strip metadata.
  • Look for images loaded dynamically from paths (file I/O) If images are loaded from disk paths at runtime inside a NetLogic, ensure the operation is in a LongRunningTask, not in Start(). Disk reads from slow or network-mounted paths can freeze the behavior thread.
Phase 6 · Critical

ExportMethod & UI Thread Freezes

⚠️ An [ExportMethod] called from a button click runs on the UI thread. Any blocking code (file I/O, database, Thread.Sleep) freezes the entire UI until it returns.
  • Identify all [ExportMethod]s triggered by UI interactions In each NetLogic, list methods decorated with [ExportMethod]. Note which ones are called from button clicks, mouse events, or UI commands — these run on the UI thread.
  • Ensure UI-triggered ExportMethods return immediately Move any work that takes more than a few milliseconds into a LongRunningTask started inside the ExportMethod. The ExportMethod itself must return within milliseconds to release the UI thread.
  • Check PeriodicTask interval vs. work duration If a PeriodicTask's work takes longer than its interval, tasks pile up. Ensure the work inside the task always finishes well within the interval period.
  • Dispose all tasks and observers in Stop() Failing to dispose LongRunningTask, PeriodicTask, and RemoteVariableSynchronizer in Stop() leaves background threads running after navigation. This leaks resources and degrades long-session performance:
    myTask?.Dispose(); myObserver?.Dispose();
Phase 7

Project Setup & Architecture

  • Review page navigation model (PanelLoader vs. full page) Loading a large page in a small PanelLoader area on every navigation adds unnecessary overhead. Consider splitting very large pages into smaller sub-pages loaded on demand.
  • Avoid loading the same large page repeatedly If a page is navigated back to often, consider keeping it resident (avoid tearing it down and reloading it). Check if the navigation pattern causes repeated Start()/Stop() cycles unnecessarily.
  • Check target hardware CPU and RAM against project complexity FTOptix performance thresholds depend on the device. A project that runs fluently on a PC may be slow on an OptixPanel with limited CPU. No universal node count limit exists — measure on the target hardware.
  • Verify no DesignTime NetLogic is accidentally invoked at runtime DesignTime NetLogics don't run Start() at runtime and are harmless if left untouched. However, if a UI element accidentally calls a DesignTime [ExportMethod], the assembly is reloaded from disk on every call — causing severe slowdown. Remove DesignTime NetLogics from deployed projects or ensure no UI bindings point to them. NetLogic Lifecycle CheatSheet
